Views and Quality Objectives of Software Construction

Multi-dimensional software views(软件构造的多维度视图)

Build-time Views

(1) Build-time, moment, and code-level view

AST

半结构化：近乎自然语言的风格+遵循特定的编程语法
前者：方便程序员
后者：方便编译器

AST：彻底结构化，将源代码变为一棵树，对树做各种操作==对源代码的修改

(2) Build-time, period, and code-level view

Code churn 代码变化:

Code churn is defined as lines added, modified or deleted to a file from one version to another.

(3) Build-time, moment, and component-level view

Library

开发者像使用编程语言指令一样使用库中的功能
System.out.println(“Hello World”); :blush:

库的来源：
操作系统提供的库
编程语言提供的库
第三方公司提供的库
你自己积累的库

编程时和build时，需告诉IDE和JVM在哪里寻找某些库

Two different approaches of integrating a library into an executable program:
–Static linking (静态链接)
–Dynamic linking (动态链接)

Static linking(静态链接)

In static linking, a library is a collection of individual object files.
库被拷贝进入代码形成整体，执行的时候无需提供库文件

Static linking happens in build time(静态链接发生在构造阶段)
–End up with a single executable program to be loaded onto the target machine.
–After the final executable program has been created, it’s impossible to separate the program from its libraries.

(4)Build-time, period, and component-level view

Version Control System (VCS)
Evolution Graph (of a SCI or a Software)

Evaluation graph describes the development process and phases of a software.（评估图描述了软件开发过程和阶段的发展情况。）

Versioning

在软件开发中，通常采用“major.minor.patch”版本号格式来标识软件的版本。这个版本号格式包含三个数字，每个数字之间用一个点号（.）隔开，它们分别代表主版本号（major）、次版本号（minor）和修订号（patch）。每个数字的含义如下：

主版本号（major）：这个数字代表软件的重大更新，通常是功能的重大改变或是架构的变化。当主版本号增加时，表明软件的核心部分已经发生了重大变化，可能导致不兼容性的问题。

次版本号（minor）：这个数字代表软件的次要更新，通常是功能的增加或是改进，但是不会影响软件的整体架构。当次版本号增加时，表明软件的某些功能得到了增强，但整体结构没有变化。

修订号（patch）：这个数字代表软件的修复和更新，通常是修复已知的漏洞或者bug。当修订号增加时，表明软件进行了一些小的修复和更新，但是没有增加新的功能或者改变整体结构。

通常，软件的版本号是以递增的方式进行更新的。例如，从版本号1.0.0升级到1.0.1表示一个修复，从1.0.0升级到1.1.0表示一个较小的更新，从1.0.0升级到2.0.0表示一个重大的更新。

Runtime Views

Native Machine Code（原生机器码）

原生机器码（Native Machine Code）是计算机可以直接执行的二进制指令，它是由CPU能够识别和执行的一系列指令。它是计算机硬件能够理解和执行的最低级别的指令集，每个指令包含一组二进制代码，用于执行特定的操作，例如数据移动、算术运算、逻辑运算、控制流程等。

原生机器吗通常使用汇编语言来编写，因为汇编语言可以直接映射到机器码。

Fully Interpreted Execution（程序完全解释执行）

程序完全解释执行（Fully Interpreted Execution）是一种编程语言的实现方式，它在运行程序时逐行解释代码，而不是将代码编译为机器语言或字节码。在程序完全解释执行的模式下，程序的每一行代码都会被解释器逐行解释并执行。
The runtime system loads the entire source code into memory and interprets it (such as BASIC, UNIX shell, etc)

Interpreted Byte Codes (解释型字节码)

解释型字节码是一种中间代码形式，它是源代码编译成可执行代码的中间步骤。与直接将源代码编译成本地机器代码不同，解释型字节码需要一个解释器来执行代码。解释器会逐行读取字节码，并将其翻译成机器代码并执行。这种方式可以实现跨平台的应用程序，因为解释器可以在不同的操作系统和硬件上运行。

Java Virtual Machine (JVM)

Perl or Python: they are interpreted rather than compiled, but use byte codes at runtime.

Dynamic linking (动态链接)

-库文件不会在build阶段被加入可执行软件，仅仅做出标记。
-程序运行时，根据标记装载库至内存。
-发布软件时，记得将程序所依赖的所有动态库都复制给用户。

Advantages
–It’s possible to upgradeto a newer version of a library (adding features or fixing bugs), without needing to re-create the executable program.
–Many operating systems can optimize their memory usage by loading only a single copy of the library into memory, yet sharingit with other programs that require that same library.

动态链接与静态链接的区别

动态链接和静态链接是在编译和链接代码时使用的两种不同的方法。

静态链接是指在编译时将所有代码和库文件链接成一个单独的可执行文件，使得执行文件包含了所有的依赖关系和函数。这种方式在运行时不需要额外的依赖库，但是会导致可执行文件变得较大，同时在多个程序使用同一库时也会产生重复的代码，浪费系统资源。

动态链接是指在运行时将程序所需的库文件动态链接到内存中，使得多个程序可以共享同一份库文件，避免了重复的代码和浪费系统资源的问题。这种方式会产生一个外部的库文件，程序需要在运行时检查库文件是否存在，如果不存在则会发生链接错误。

总的来说，静态链接比较适合小型应用程序，而动态链接则更适合大型程序，特别是在多个程序共享同一库文件的情况下。

(5) Run-time, moment, and code-level view

代码快照图

描述程序运行时内存里变量层面的状态
Fine-grained(细粒度) states of a program.

Memory dump (内存信息转储)

“Memory dump” 是指将计算机的内存中的所有数据和程序状态记录下来并存储到硬盘上的过程。

举个例子来说，就好像你在玩一个电脑游戏时，游戏突然崩溃了，而你需要向开发商或技术支持团队报告这个问题。在这种情况下，你可以通过创建一个”memory dump”来捕获崩溃时的状态和数据，并将其保存到文件中。然后，开发商或技术支持团队可以分析这个文件，找出问题出在哪里，以便修复它。

在计算机安全领域，”memory dump”也被用于分析恶意软件或攻击事件。当系统遭受攻击或感染病毒时，安全专家可以创建一个”memory dump”来获取有关攻击者或病毒的详细信息，以便进行进一步分析和调查。

总的来说，”memory dump”是一种非常有用的工具，它可以帮助人们在计算机系统出现问题时更好地理解和解决问题，同时也可以帮助安全专家追踪和分析恶意软件和攻击事件。

Software construction: Transformation between views(软件构造的阶段划分、各阶段的构造活动)

Quality properties of software systems(软件的内部/外部的质量指标)

External quality factors affect users外部质量因素影响用户
Internal quality factors affect the software itself and its developers内部质量因素影响软件本身和它的开发者
External quality results from internal quality外部质量取决于内部质量

Summary

Three dimensions of describing a software system:

–By phases(阶段划分): build-and run-time views(构造时/运行时视图)
–By dynamics(动态性划分): moment and period views(时刻/阶段视图)
–By levels(构造对象的层次划分): code and component views(代码/构件视图)

Quality properties of software systems

–External vs. internal quality factors
–Important external quality factors(外部质量指标、内部质量指标)
–Tradeoff between quality factors(质量指标之间的折中)

Five key quality objectives of software construction（软件构造的五个关键质量目标）

–Easy to understand: elegant and beautiful code / understandability(易理解)
–Ready for change: maintainability and adaptability(可变化性)
–Cheap for develop: design for/with reuse: reusability(开发成本)
–Safe from bugs: robustness(安全性)
–Efficient to run: performance(性能)

外部质量因素： correctness（正确性）,robustness（健壮性）,extendibility（可扩展性）,reusability（可复用性）,compatibility（兼容性）除此之外还有可以执行，可修复性，经济性等等

内部的质量因素： 代码行数、低耦合、可读性、可理解性、清洁度等
